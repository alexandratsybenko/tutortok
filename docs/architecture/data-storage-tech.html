Технологии хранения данных

#Технологии хранения данных

1. Определение сущностей, которые необходимо хранить​

Сущности:​

  1. Пользователь

  2. Профиль репетитора

  3. Видео

  4. Чат

  5. Сообщение

  6. Урок

  7. Платеж

  8. Документ ****

  9. Лайк

  10. Уведомление

  11. Событие аналитики

Пользователь

Атрибуты:

  • id (UUID) - уникальный идентификатор

  • email (string) - email, уникальный

  • password_hash (string) - хэш пароля

  • first_name (string) - имя

  • last_name (string) - фамилия

  • avatar_url (string, nullable) - ссылка на аватар в S3/CDN

  • role (enum: STUDENT, TUTOR, ADMIN) - роль в системе

  • email_verified (boolean) -подтвержден ли email

  • created_at (timestamp) - дата регистрации

  • updated_at (timestamp) - дата последнего изменения

Акторы: Ученик, Репетитор, Администратор

Характер данных: Транзакционные (OLTP). Строгая консистентность (уникальность email). Чтение при каждом логине.

Профиль репетитора

Атрибуты:

  • id (UUID) - связан с User.id (связь: один к одному)

  • nickname (string) - уникальный никнейм для URL

  • is_verified (boolean) - агрегирующий статус верификации

  • language (array[string]) - основной язык преподавания

  • specializations (array[string]) - список специализаций

  • price_per_hour (decimal) - стоимость часа

  • experience_years (integer) - стаж в годах

  • rating (float) - средний рейтинг (денормализован)

  • about (text) - текст «О себе»

  • video_trailer_url (string) - ссылка на видео-визитку

  • created_at (timestamp)

  • updated_at (timestamp)

Акторы: Репетитор (CRUD), Ученик (R - просмотр), Администратор (R U- модерация и обновление)

Характер данных: транзакционные (OLTP) с преобладанием чтения. Профили репетиторов часто запрашиваются для отображения в ленте и на странице профиля, но редактируются редко.

Видео (видеовизитки + бесплатные уроки)

Атрибуты:

  • id (UUID)

  • tutor_id

  • title (string, nullable) - название (только для уроков)

  • description (text, nullable) - описание

  • preview_url (string) - ссылка на превью в S3/CDN

  • video_url (string) - ссылка на видео в S3/CDN

  • duration (integer) - длительность в секундах

  • type (enum: TRAILER, LESSON) - тип видео

  • status (enum: PENDING_MODERATION, APPROVED, REJECTED)

  • created_at (timestamp)

  • updated_at (timestamp)

Акторы: Репетитор (C, D), Ученик (R), Администратор (U - модерация)

Характер данных: Метаданные - транзакционные, файлы - потоковые (Object Storage).

Чат

Атрибуты:

  • chat_id (UUID)

  • student_id

  • tutor_id

  • created_at (timestamp)

  • last_message_at (timestamp) - денормализовано для сортировки списка чатов

Акторы: Ученик (C - создание), Репетитор (R - чтение)

Характер данных: Транзакционные (OLTP)

Сообщение

Атрибуты:

  • id (UUID)

  • chat_id

  • sender_id

  • text (text)

  • created_at (timestamp)

  • edited_at (timestamp, nullable

  • is_edited (boolean)

  • is_read (boolean)

Акторы: Ученик, Репетитор (CRUD)

Характер данных: транзакционные (OLTP, требуется постоянная запись)

Урок

Атрибуты:

  • id (UUID)

  • chat_id

  • student_id

  • tutor_id

  • scheduled_at (timestamp) - запланированное время

  • duration_minutes (integer)

  • price (decimal)

  • status (enum: AWAITING_PAYMENT, PAID, CONFIRMED, COMPLETED, CANCELED, DISPUTED)

  • telemost_link (string, nullable) - ссылка на Яндекс.Телемост

  • created_at (timestamp)

  • expires_at (timestamp) - для автоматической отмены при неоплате

Акторы: Ученик (оплата, подтверждение), Репетитор (подтверждение), Система (таймеры)

Характер данных: Транзакционные, важно соблюдать ACID

Платеж

Атрибуты:

  • id (UUID)

  • lesson_id (FK → Lesson)

  • student_id (FK → User)

  • amount (decimal)

  • status (enum: PENDING, HELD, CAPTURED, REFUNDED)

  • external_payment_id (string) - ID транзакции в ЮKassa

  • created_at (timestamp)

  • updated_at (timestamp)

Акторы: Ученик (C - инициирование), Система (U - webhook)

Характер данных: Транзакционные, финансово-критичные (ACID)

Документ (репетиторы отправляют документы на проверку для подтверждения личности)

Атрибуты:

  • id (UUID)

  • tutor_id (FK → TutorProfile)

  • document_url (string) - приватная ссылка на защищенное S3

  • status (enum: PENDING, APPROVED, REJECTED)

  • admin_comment (text, nullable) - причина отказа

  • uploaded_at (timestamp)

  • reviewed_at (timestamp)

Акторы: Репетитор (C), Администратор (R/U)

Характер данных: Транзакционные

Лайк

Атрибуты:

  • student_id (FK → User)

  • tutor_id (FK → TutorProfile)

  • created_at (timestamp)

Акторы: Ученик (C/D)

Характер данных: Транзакционные (постоянная запись). Для чтения допустима Eventual Consistency.

Уведомление

Атрибуты:

  • id (UUID, PK)

  • user_id (FK -> User)

  • type (enum: LESSON_REMINDER, MODERATION_RESULT, NEW_MESSAGE, PAYOUT)

  • title (string)

  • body (text)

  • is_read (boolean)

  • created_at

Акторы: Система (C), ученик/репетитор (R/U - чтение/отметка прочитанным).

Характер данных: Транзакционные. Чаще запись. Требуется хранение данных (для просмотра последних уведомлений). Формат уведомлений непостоянный.

Данные для аналитики

Сущность, необходимая для развития платформы (версия 2.0+).

Поток событий о действиях пользователей (просмотры видео, клики по профилям, отправка сообщений).

Атрибуты:

  • event_id (UUID)

  • event_type (string)

  • user_id (UUID)

  • payload (JSON) - детали события

  • timestamp (timestamp)

Акторы: Система (C - запись), Аналитик / BI-система (R - пакетное чтение).

Характер данных: Аналитические (OLAP). Очень большой объем, потоковая запись, не требуют строгой консистентности в реальном времени, оптимизированы для пакетной обработки и агрегации.

Аанализ показал, что оптимальная архитектура хранения данных должна быть спроектирована на основе polyglot persistence (использование разных технологий хранения под конкретные задачи).

1. Реляционная СУБД

Большая часть сущностей (пользователи, профили, уроки, платежи, метаданные, чаты) имеет жесткую реляционную структуру и требует транзакционной целостности (strong consistency, ACID). Ожидаемые объемы данных на этапе MVP небольшие (до 5 ГБ), так что реляционная СУБД легко справится с нагрузкой (OLTP).

Есть сущность, которая может потребовать гибкой структуры в зависимости от типа: уведомления. Будет использован тип данных Json внутри той же РСУБД. Избавляет от необходимости разворачивать отдельную Document DB (например, MongoDB). Для бюджета и заложенного времени на разработку MVP - это оптимальный вариант.

2. Object Storage (S3-совместимое хранилище)

Бинарные файлы (видеовизитки, бесплатные уроки, сканы паспортов) обладают паттерном доступа Append, full read (запись один раз, чтение целиком). Хранение 120+ ГБ неструктурированных данных в реляционной СУБД приведет к ухудшению производительности. Поэтому файлы выносятся в облачное хранилище S3, которое обеспечивает горизонтальное масштабирование, потоковую отдачу видео, дешевое хранение, а в РСУБД сохраняются только ссылки на эти объекты.

3. Эволюционная архитектура

В рамках MVP приняты архитектурные допущения:

  • Лайки и счетчики на старте будут храниться в РСУБД. При росте платформы этот функционал будет вынесен в Key-Value хранилище (например, Redis) для обеспечения максимальной скорости и разгрузки основной базы.

  • События аналитики (OLAP-нагрузка) изолированы от транзакционных данных. В версии 2.0+ планируется выгрузка этих событий (event streaming) в Data Lake для агрегации, построения ИИ-рекомендаций для ленты видео без влияния на производительность операционной базы данных.

Таким образом, выбранный стек технологий закрывает текущие потребности пользователей платформы, укладывается в рамки MVP и имеет потенциал для горизонтального масштабирования в будущих версиях.
